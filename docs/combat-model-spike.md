# Combat Model Spike - Design Document

## Overview

This document describes the initial implementation of the combat event model, validated through the Augur firing pattern example.

The goal was to prove that:

**Ship Definition → Firing Pattern → Combat Events**

can be represented cleanly using a framework-agnostic domain model.

---

## Design Decisions

### 1. Round-Based Model

Combat is modeled as discrete rounds rather than continuous time.

**Rationale:**
- STFC combat operates in rounds
- Simplifies event generation
- Matches player mental model
- Easier to reason about and test

**Trade-off:**
- Less suitable for real-time simulation
- Acceptable for firing pattern visualization

---

### 2. Firing Schedule Types

Three schedule patterns support the majority of STFC firing behaviors:

#### Every Round
```typescript
{ type: 'every_round' }
```
Weapon fires every round (e.g., Augur beams).

#### Interval-Based
```typescript
{
  type: 'interval',
  startRound: 2,
  interval: 3
}
```
Weapon fires at regular intervals (e.g., Augur Obliterator on rounds 2, 5, 8, 11...).

#### Specific Rounds
```typescript
{
  type: 'specific_rounds',
  rounds: [1, 3, 5, 7]
}
```
Weapon fires on explicitly listed rounds. Supports irregular patterns.

**Rationale:**
- Union type provides type safety
- Extensible without breaking existing schedules
- Covers observed STFC weapon behaviors

**Future Extension Points:**
- Conditional schedules (e.g., "fire if shield depleted")
- Charge-up mechanics
- Reactive patterns (e.g., "fire when hit")

---

### 3. Shots Per Activation

Each weapon defines `shotsPerActivation` to represent multi-shot weapons.

**Example:**
- Augur beams fire 2 shots when activated
- Obliterator fires 1 shot when activated

**Rationale:**
- Single field handles burst fire
- Avoids redundant event generation
- UI/animation layer controls visual shot timing

**Note:**
This does NOT model:
- Shot spread/delay between shots in a burst
- Individual shot damage variance
- Critical hits

Those concerns belong in animation or detailed combat simulation layers.

---

### 4. Combat Event Structure

Events are simple, serializable structures:

```typescript
interface CombatEvent {
  type: 'weapon_fired' | 'round_start' | 'round_end';
  round: number;
  weaponId?: string;
  weaponName?: string;
  shots?: number;
}
```

**Rationale:**
- Framework-agnostic (no dependencies on React, UI, animation)
- Serializable for logging, replay, network transmission
- Minimal surface area
- Easy to test

**What's NOT included:**
- Timestamps (can be derived from round number)
- Damage values (belong in damage calculation layer)
- Visual effects (belong in animation layer)
- Target information (out of scope for Milestone 1)

---

### 5. Event Generation Functions

Two pure functions generate events:

#### `generateRoundEvents(ship, round): CombatEvent[]`
Returns all events for a single round.

#### `generateCombatEvents(ship, startRound, endRound): RoundEvents[]`
Returns events for a range of rounds.

**Rationale:**
- Pure functions (no side effects)
- Deterministic output
- Easy to test
- Composable

**Design Note:**
Functions accept `Ship` rather than pre-processed data structures. This keeps the API simple and allows the combat model to interpret firing schedules internally.

## Implementation Validation

### Augur Firing Pattern

The Augur example validates the model:

**Weapons:**
- Left Beam: 2 shots every round
- Right Beam: 2 shots every round
- Obliterator: 1 shot every 3 rounds starting round 2

**Expected Behavior:**

| Round | Left Beam | Right Beam | Obliterator |
|-------|-----------|------------|-------------|
| 1     | ✓ (2)     | ✓ (2)      | -           |
| 2     | ✓ (2)     | ✓ (2)      | ✓ (1)       |
| 3     | ✓ (2)     | ✓ (2)      | -           |
| 4     | ✓ (2)     | ✓ (2)      | -           |
| 5     | ✓ (2)     | ✓ (2)      | ✓ (1)       |
| ...   | ...       | ...        | ...         |

This pattern is correctly generated by the model.

## What This Model Does NOT Cover

By design, the following are **out of scope** for Milestone 1:

### Combat Mechanics
- Damage calculation
- Shield/hull interactions
- Accuracy/evasion
- Critical hits
- Damage mitigation

### Officers & Buffs
- Officer abilities
- Crew synergies
- Temporary buffs/debuffs

### Research & Upgrades
- Research bonuses
- Ship tier upgrades
- Component improvements

### Targeting & Strategy
- Target selection
- Multi-ship combat
- Formation effects

### Real-Time Simulation
- Continuous time progression
- Animation timing
- Visual effects

**Rationale:**
These concerns can be layered on top of the event model in future milestones without changing the core architecture.

## Future Extension Points

The current design supports future enhancements:

### 1. Conditional Schedules
Add new schedule types:
```typescript
{
  type: 'conditional',
  condition: 'shield_depleted',
  baseSchedule: { type: 'every_round' }
}
```

### 2. Officer Abilities
Extend `Ship` to include officer effects:
```typescript
interface Ship {
  officers?: OfficerEffect[];
}
```

Officers can modify firing schedules or event generation without changing core types.

### 3. Damage Calculation
Add optional damage details to events:
```typescript
interface CombatEvent {
  damageDealt?: number;
  damageType?: string;
}
```

### 4. Animation Metadata
Events can carry animation hints without coupling to specific animation systems:
```typescript
interface CombatEvent {
  visualHints?: {
    effectType?: string;
    duration?: number;
  };
}
```

### 5. Multi-Ship Combat
Extend event generation to accept multiple ships:
```typescript
generateCombatEvents(
  attacker: Ship,
  defender: Ship,
  rounds: number
): CombatEvent[]
```

## Architectural Observations

### What Worked Well

1. **Separation of Concerns**
   - Ship model defines "what fires when"
   - Combat model interprets schedules and generates events
   - No UI coupling

2. **Type Safety**
   - Union types for firing schedules prevent invalid configurations
   - TypeScript catches misuse at compile time

3. **Testability**
   - Pure functions with deterministic outputs
   - Easy to write unit tests
   - No mocking required

4. **Simplicity**
   - Minimal concepts: Ship, Weapon, Schedule, Event
   - Easy to explain and understand
   - Low cognitive overhead

### Potential Issues

1. **Performance at Scale**
   - Generating events for 1000+ rounds may become slow
   - Mitigation: Cache or lazy-generate events

2. **Complex Patterns**
   - Some ships may have irregular patterns not fitting the three schedule types
   - Mitigation: Add new schedule types as needed

3. **Event Ordering**
   - No explicit ordering for weapons firing in the same round
   - Current behavior: array order in `ship.weapons`
   - May need explicit priority field later

### Lessons Learned

1. **Start Simple**
   - Initial over-engineered designs were rejected
   - Three schedule types cover 90% of patterns
   - Additional complexity can be added when needed

2. **Framework Agnostic Pays Off**
   - No React/UI dependencies make testing trivial
   - Can use this model in Node.js scripts, web apps, or future tools

3. **Event Model is Powerful**
   - Events decouple "what happens" from "how it's displayed"
   - Future combat simulators can consume the same events
   - Replay, logging, and debugging become trivial

## Conclusion

The combat event model successfully demonstrates that STFC firing patterns can be represented cleanly using simple, type-safe domain models.

The design is:
- **Simple** - minimal concepts, easy to understand
- **Extensible** - supports future enhancements without breaking changes
- **Framework-agnostic** - reusable across applications
- **Testable** - pure functions with deterministic outputs

The Augur example validates the core architecture and provides a foundation for future development.
